# OpenSkills Interop Analysis for OpenTeams

## Overview

This document analyzes [OpenSkills](https://github.com/numman-ali/openskills) — a universal skills loader for AI coding agents — and identifies concrete ways its approach can advance OpenTeams' goal of interoperable multi-agent team coordination.

## What OpenSkills Does

OpenSkills makes Anthropic's SKILL.md format work across any AI coding agent (Cursor, Windsurf, Aider, etc.). It has three layers:

1. **Installation** — fetches skills from GitHub repos, local paths, or private repos via shallow git clones
2. **Storage** — organizes skills as static file directories (`.agent/skills/` for universal, `.claude/skills/` for Claude-specific)
3. **Loading** — generates an AGENTS.md file with XML metadata that agents parse to discover and invoke skills on demand

The key design choices:
- **Static files over servers** — no MCP server dependency, just SKILL.md files on disk
- **Progressive disclosure** — agents see a skill catalog (name + description), then load full content only when needed
- **Priority resolution** — project-local universal > global universal > project-local Claude > global Claude
- **Stateless invocation** — each `openskills read <name>` call is independent

## Where OpenTeams and OpenSkills Intersect

OpenTeams already has two generators that produce agent-consumable output:

| Generator | Output | Purpose |
|-----------|--------|---------|
| `generateSkillMd()` | Single SKILL.md | Encodes team topology, communication, and CLI usage for "naive" agents |
| `generateAgentPrompts()` | Per-role prompts | Role-specific context with subscriptions, emissions, and spawn rules |

OpenSkills provides the **distribution and discovery infrastructure** that these generators currently lack. OpenTeams can generate the content; OpenSkills can deliver it to any agent.

## Concrete Interoperability Opportunities

### 1. Package OpenTeams team templates as installable skills

OpenTeams' `generateSkillMd()` already produces content in the right shape. The gap is packaging:

```
openteams-coordination/
├── SKILL.md          # Generated by generateSkillMd() with YAML frontmatter
├── references/
│   └── team.yaml     # The source template
├── scripts/
│   └── setup.sh      # Installs openteams CLI, loads template
└── assets/
    └── topology.md   # Visual diagram of team structure
```

A team author would:
1. Define a team template (`team.yaml`, roles, prompts)
2. Run `openteams generate skill ./templates/my-team -o ./skills/my-team/`
3. Publish the skill directory to a Git repo
4. Any agent installs it: `npx openskills install org/team-skills`

### 2. Use AGENTS.md as the universal team roster

OpenSkills generates an `<available_skills>` XML block in AGENTS.md. OpenTeams could extend this pattern to advertise team roles and communication channels, not just skills:

```xml
<skills_system priority="1">
<available_skills>

<skill>
<name>openteams-planner</name>
<description>Planner role in the self-driving team. Manages task breakdown, assigns work, monitors progress. Subscribes to: task_updates, work_coordination. Can emit: TASK_CREATED, WORK_ASSIGNED.</description>
<location>project</location>
</skill>

<skill>
<name>openteams-grinder</name>
<description>Worker role in the self-driving team. Claims and executes tasks. Subscribes to: work_coordination (WORK_ASSIGNED). Can emit: WORKER_DONE.</description>
<location>project</location>
</skill>

</available_skills>
</skills_system>
```

When an agent reads a team role skill, it gets the full prompt with CLI commands, communication patterns, and spawn rules — everything `generateAgentPrompts()` currently produces.

### 3. Progressive disclosure for team context

OpenSkills' progressive disclosure model solves a real problem for OpenTeams: team context can be large (topology + communication + all role prompts), and dumping it all into an agent's context upfront is wasteful.

With OpenSkills' pattern:
- **Catalog phase**: Agent sees a compact list of team roles with one-line descriptions
- **Load phase**: Agent runs `openskills read openteams-planner` only when it needs to act in that role
- **Stateless**: Each role load is independent — agents can switch roles or load multiple

This is better than OpenTeams' current approach where `generateSkillMd()` produces one monolithic document with everything in it.

### 4. Multi-agent, multi-tool team bootstrapping

The most valuable interop scenario: a team template that works across agent platforms.

Current state: OpenTeams generates prompts optimized for Claude's Task tool. Non-Claude agents can't use these.

With OpenSkills integration:
1. Team template defines roles in `team.yaml` (agent-agnostic)
2. OpenTeams generates per-role SKILL.md files (one per role)
3. OpenSkills distributes them to any agent platform
4. Each agent reads its role's SKILL.md, gets CLI commands for openteams coordination
5. The SQLite database remains the shared coordination layer

The agents don't need to understand each other's internals — they communicate through `openteams` CLI commands and the shared database.

### 5. Skill composition: team skills + domain skills

OpenSkills supports installing multiple skills. An agent could load both team coordination skills (from OpenTeams) and domain-specific skills (from other repos) simultaneously:

```bash
# Install team coordination
npx openskills install myorg/team-templates

# Install domain skills
npx openskills install anthropics/skills

# Agent sees both in AGENTS.md and can use either
```

This maps well to OpenTeams' extensible template schema — the `macro_agent` field and other extension fields could reference domain skills that each role should load.

## What OpenTeams Would Need to Build

### A. `generate skill-package` CLI command

A new CLI command that converts a resolved template into an OpenSkills-compatible skill package:

```bash
openteams generate skill-package ./templates/self-driving -o ./skills/
```

This would:
1. Load and resolve the template
2. Generate a SKILL.md with proper YAML frontmatter (`name`, `description`, `context`)
3. Copy the source template into `references/`
4. Generate per-role SKILL.md files in subdirectories
5. Optionally generate a `scripts/bootstrap.sh` that runs `openteams template load`

### B. YAML frontmatter for generated SKILL.md

OpenSkills requires YAML frontmatter. The current `generateSkillMd()` output lacks this. Adding it:

```markdown
---
name: self-driving-team
description: Autonomous development team with planner, grinder, and judge roles
context: multi-agent-coordination
---

# Team: self-driving
...
```

### C. Role-specific skill packages

Instead of one monolithic SKILL.md, generate one skill per role:

```
self-driving-team/
├── SKILL.md                    # Team overview + role catalog
├── roles/
│   ├── planner/
│   │   └── SKILL.md            # Full planner prompt + context
│   ├── grinder/
│   │   └── SKILL.md            # Full grinder prompt + context
│   └── judge/
│       └── SKILL.md            # Full judge prompt + context
└── references/
    └── team.yaml               # Source template
```

### D. AGENTS.md sync integration

A command or hook that syncs OpenTeams team roles into the project's AGENTS.md in the format OpenSkills expects:

```bash
openteams generate agents-md self-driving >> AGENTS.md
# or
npx openskills sync  # picks up OpenTeams skills from .agent/skills/
```

## Architectural Alignment

| Concern | OpenTeams | OpenSkills | Integration Point |
|---------|-----------|------------|-------------------|
| Team definition | `team.yaml` templates | — | OpenTeams defines, exports as skills |
| Agent instructions | `generateAgentPrompts()` | SKILL.md files | Prompts become SKILL.md content |
| Discovery | Database queries | `findAllSkills()` + AGENTS.md | AGENTS.md as universal catalog |
| Distribution | Local filesystem | Git repos + install CLI | `openskills install` for sharing |
| Runtime coordination | SQLite + CLI commands | — | OpenTeams CLI remains the runtime layer |
| Context efficiency | Monolithic SKILL.md | Progressive disclosure | Per-role skills loaded on demand |
| Agent compatibility | Claude Task tool | Any agent | OpenSkills bridges the gap |

## Key Insight

OpenSkills and OpenTeams are complementary, not competing:

- **OpenTeams** handles the runtime: team state, task queues, messaging, signal routing, agent lifecycle — all backed by SQLite with CLI access
- **OpenSkills** handles the distribution: packaging agent instructions as portable SKILL.md files, discovering them across projects, and loading them into any agent platform

The combination enables a workflow where team templates are **authored once** in OpenTeams' YAML format, **distributed** through OpenSkills' git-based installation, and **executed** by any agent that can run shell commands — regardless of whether it's Claude Code, Cursor, Windsurf, or something else entirely.

The shared database + CLI pattern means agents don't need a common protocol beyond "run this shell command." OpenSkills ensures they all receive the right instructions for their role.
